description: 변경사항을 자동 커밋·푸시하고 main 대상 PR을 만들어 머지까지 처리
---

현재 작업 브랜치의 변경사항을 끝까지 배송(ship)한다. 사용자에게 매번 확인받지 말고 아래를 자동으로 진행한다. 추가 지시가 있으면 참고: $ARGUMENTS

## 절차

1. **상태 확인**: `git status`, `git diff origin/main...` 로 변경 내용을 파악한다.
   - 변경이 없으면 "배송할 변경사항이 없다"고 알리고 종료.
   - 현재 브랜치가 `main`이면 **중단**하고 경고한다 (main 직접 커밋 금지). 새 `feat/…` 브랜치를 만들 것을 제안.

2. **Unity 안전 점검**:
   - `Library/`, `Logs/`, `Temp/`, `UserSettings/` 같은 무시 대상이 스테이징되려 하면 제외한다.
   - 새 에셋이 있으면 짝이 되는 `.meta` 파일이 함께 있는지 확인한다. `.meta` 누락 시 경고.

3. **커밋**: 변경 내용을 요약해 커밋 메시지를 자동 작성한다.
   - 형식: `<타입>: <한글 요약>` (타입: feat/fix/chore/docs/refactor/art)
   - **`Co-Authored-By:` 줄은 절대 넣지 않는다.**
   - 논리적으로 나눌 수 있으면 여러 커밋으로 쪼갠다.

4. **푸시**: `git push -u origin <현재 브랜치>`.

5. **PR**: `gh pr create --base main` 로 PR을 만든다 (이미 있으면 재사용).
   - 제목 = 대표 커밋 메시지, 본문 = 무엇을/왜 바꿨는지 3~5줄 한글 요약 + 관련 로드맵 단계.
   - 관련 이슈가 있으면 PR 본문에 `Closes #N` 을 넣어 머지 시 자동으로 닫히게 한다.
   - PR 본문 끝에 아래 한 줄만 붙인다: `🤖 Generated with [Claude Code](https://claude.com/claude-code)`

6. **머지**: CI가 없거나 통과 상태면 `gh pr merge --squash --delete-branch` 로 자동 머지한다.
   - 충돌이나 실패가 있으면 머지하지 말고 원인을 사용자에게 한국어로 간단히 보고한다.

7. **이슈 정리**: 이번 작업에 대응하는 GitHub 이슈가 있으면 `Closes #N`으로 닫히게 하거나, 없으면 필요 시 만든다.

## 마무리 보고

무엇을 커밋했고, PR/머지 결과 링크와 상태를 한국어로 3줄 이내로 요약한다.
